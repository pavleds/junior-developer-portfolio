The problem

A café order crosses several roles: the cashier enters it, the kitchen prepares it, staff deliver it, and the manager reviews the day. If those steps are disconnected, mistakes and delays become more likely.

Kafa is a front-end prototype that makes that workflow visible. It intentionally avoids pretending to be a fiscal cash register or payment processor.

The approach

I treated each staff role as a focused workspace while keeping all data connected. Sending an order from the register immediately creates a kitchen ticket and updates reporting.

Fast register

Categories, search, large product targets, quantities, table choice, and clear totals reduce interaction cost.

Visible status

Kitchen tickets move through waiting, preparing, and ready columns.

Live menu

Availability controls instantly prevent sold-out products from being ordered.

Useful reporting

Revenue, averages, popular items, activity, and CSV export make the demo feel connected.

Technical details

Shared state

Products, the current cart, order history, service type, and sequence numbers are represented as JavaScript data. Each view renders from that shared state.

Persistence

Menu changes and orders are serialized into localStorage. The app restores them after refresh, while a reset action safely returns everything to the original demonstration state.

Workflow logic

Orders begin as waiting, advance to preparing, and finish as ready. Reports calculate directly from order history, so every register action has a visible downstream effect.

Safety boundaries

The interface clearly says it has no real payments or fiscal receipts. A production version would require authentication, server-side validation, a database, device synchronization, backups, security review, and compliant fiscal integration.

What I learned

  • A multi-view application becomes easier to reason about when every screen renders from one predictable data model.
  • Operational interfaces need large targets, unmistakable states, and recovery actions.
  • Reports are only trustworthy when they derive from the same data as the workflow.
  • A prototype should communicate its limitations as clearly as its capabilities.

The outcome

A persistent café operations prototype spanning register, kitchen, menu management, bilingual labels, and daily reporting. It demonstrates a complete workflow while maintaining an honest boundary between portfolio software and production POS infrastructure.
